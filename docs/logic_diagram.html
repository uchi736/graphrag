GraphRAG ロジック詳細図解

PDF/テキストから ナレッジグラフ (Neo4j) と ベクトルDB (PGVector) を構築し、 グラフ探索とハイブリッド検索を統合して高精度な質問応答を行う Graph-First RAG システム。 本ドキュメントは graphrag_core/ の全ロジックをコードに基づいて図解する。

構築 Build 後処理 Post-Processing 検索 Retrieval 回答生成 Generation データ / 定数
2つの大きなフェーズ
① 構築フェーズ（オフライン） 文書 → チャンク → LLMでトリプル抽出 → Neo4j保存 → 後処理（統合・正規化・enrichment）→ PGVector化。build_kg.py / UI「構築」から実行。
→
② 検索フェーズ（オンライン） 質問 → エンティティ抽出 ∥ 文書検索（並列）→ N-hopグラフ探索 → cross-encoderリランク → コンテキスト統合 → LLM回答。retrieval/pipeline.py。
設計思想: 「広く取得 → 狭くリランク」を全レイヤ（文書・グラフパス・KGチャンク）で徹底。DB側は extraction_count/pagerank/BM25/ベクトルでスコア付けしてから取得するため、無作為な LIMIT 切り捨てが起きない。最終段は cross-encoder（bge-reranker-v2-m3、最強レバー +11.8pt）で意味的に絞り込む。

BUILD1. 構築パイプライン全体 scripts/build_kg.py

build_knowledge_graph(input_dir, extensions, fresh) がマスターシーケンス。 ファイル探索から graph.json エクスポートまでを一気通貫で実行する。

Aファイル探索・読込input_dir.glob("**/*.{ext}") を拡張子ごとに再帰探索し sorted(set(...)) で重複排除。PDFは load_pdf_text、txt/mdはUTF-8読込。source=ファイル名のみ（フルパスでない）。
▼
Bチャンキング + ハッシュ重複排除create_markdown_chunks(..., chunk_size=1024, chunk_overlap=100)。各チャンクに make_chunk_id(source, content)=SHA256 を付与し metadata["id"] に格納。文書横断の重複排除はしない（文書スコープ削除の安全性優先）。
▼
CNeo4j初期化Neo4jGraph(enhanced_schema=True)。--fresh 時は MATCH (c:ProcessedChunk) DELETE c（再開マーカーのみクリア）。
D再開フィルタ既存 ProcessedChunk.hash を取得し、未処理チャンクのみ pending_chunks に。すべて処理済なら skip_graph_build=True。
▼
ELLMTransformer設定LLMGraphTransformer(allowed_nodes=["Term"], allowed_relationships=12関係, strict_mode=True)。vLLMは ignore_tool_usage=True + additional_instructions、Azure等はカスタムprompt（KG_SYSTEM_PROMPT）。
▼
F並列チャンク処理ThreadPoolExecutor(max_workers=KG_BUILD_WORKERS=4)。各チャンク: ①トリプル抽出 → ②add_graph_documents(include_source=True) → ③attach_source_chunks → ④ProcessedChunkマーク → ⑤Document.source設定。
▼
HPGVector保存接続timeout 30s。uniqueness/schema/HNSWインデックス確保。日本語トークン化（Sudachi→tokenized_content、BM25用）+ エンティティベクトル化（EntityVectorizer）。
▼
I後処理チェーン（順序が重要） consolidate → 参照グラフ+照応解決 → 条件付き事実(opt) → 辞書適用(opt) → enrichment → スキーマ刻印
§4 consolidate→ §5 references→ §6 conditions→ §7 dictionary→ §8 enrichment
▼
Jgraph.json エクスポートNetworkX node_link_data 互換の {"graph":{directed, multigraph, nodes, edges}}。ProcessedChunk/Chunk/ハッシュIDノードを除外。
なぜ後処理の順序が固定なのか: ① consolidate を最初に走らせ、ノードマージ後に下流が正規ノードへ適用されるようにする。 ② dictionary は enrichment より前（search_keys が aliases/canonical_form を取り込むため）。 ③ enrichment は最後（pagerank は最終グラフ構造に依存、エッジ移動で mention_count も変わる）。

BUILD2. Markdown 2段階チャンキング text/chunking.py

create_markdown_chunks(docs, chunk_size=1024, chunk_overlap=100, add_breadcrumb=True)。 見出し構造を保ちつつ、埋め込み・BM25で検索可能な「パンくず付き」チャンクを生成する。

Stage 1 — 構造分割
  • _normalize_markdown_headers: ##見出し→## 見出し にスペース補正
  • _extract_h1_title: H1 をメタデータとしてのみ抽出（分割境界にしない）
  • MarkdownHeaderTextSplitter: H2/H3 のみで分割、strip_headers=False で本文に見出しを残す
→
Stage 2 — 文字再分割（len > 1024 のみ）
  • RecursiveCharacterTextSplitter(1024, 100)
  • 日本語対応セパレータ: ["\n\n","\n","。","、"," ",""]
  • メタデータに source + h2/h3 + h1 をマージ
パンくず注入 _prepend_breadcrumb

文字再分割後の2つ目以降のサブチャンクは見出しを失う。そこで本文先頭に [h1 > h2 > h3]\n を本文として注入する（メタデータだけでなく）。これにより セクション文脈が埋め込みベクトルにも BM25 インデックスにも乗る。

[ガス軸受設計 > 2. 軸受形式 > 2.1 動圧型]+本文テキスト…→埋め込み/BM25対象
注意: モジュール docstring は「500文字」と書いてあるが、実際の既定は chunk_size=1024（コード既定・build_kg 呼び出しとも 1024）。コード値が正。

BUILD3. KG抽出（LLMGraphTransformer）prompts.py / schema.py

チャンクごとに LLM がエンティティ（Term）と関係トリプルを抽出。スキーマは外部JSONで差し替え可能。

ノード抽出基準 KG_SYSTEM_PROMPT

  • 抽出する: 技術用語・概念・固有名詞・プロセス名・規格名
  • 抽出しない: 一般名詞（こと/もの/方法）・代名詞・動詞
  • 値はノードにしない: 「210円」「53」「令和6年度」「約10時間」。値は関係の対象を概念名にする（例: 牛乳 -[HAS_ATTRIBUTE]→ 価格）

デフォルト12関係 get_allowed_relations()

IS_ABELONGS_TO_CATEGORYPART_OF HAS_STEPHAS_ATTRIBUTERELATED_TO AFFECTSCAUSESDEPENDS_ON APPLIES_TOOWNED_BYSAME_AS
RELATED_TO は他に適切な関係がない場合の最終手段。SHARED_SCHEMA_PATH で外部JSON（例: 8ノード+23関係）に差し替え可能。適用中スキーマは :SchemaMeta ノードに刻印。
エンティティ判定の中心述語 entity_node_predicate(var) schema.py:43-72

「これは本物の検索対象エンティティか？」を判定するCypher述語。ラベル非依存（ノードタイプ差し替え対応）。システム全域（enrichment・prune・pagerank・辞書・検索）で再利用される。次を全て除外:

id が NULL :ProcessedChunk :SchemaMeta :GraphProvenance ハッシュID [0-9a-f]{32,} is_value=true is_anaphor=true :CondFact / :Cond（条件機能ON時）
条件機能OFF時は生成文字列がバイト単位で不変（高頻度再利用関数のため）。consolidate が付けるフラグ(is_value/is_anaphor)がこの除外を直接広げる。

POST4. KG統合 consolidate graph/consolidate.py

consolidate_post_build(graph, similarity_threshold=0.9) が5サブステップを順に実行し統計を返す。 LLM抽出の生KGにある構造的ノイズ（値ノード・型分裂・かな揺れ・逆関係・照応）を機械的に解消する。

  1. ① flag_value_nodes — 値ノードのフラグ付け
    n.id が下記正規表現に一致するノードに is_value=true（削除せず可逆）。トラバーサル・entity vector・enrichmentから除外されるがパス終端(葉)としては許可（数値質問の救済）。実測5,715件。
    • 数値・単位: ^[0-9０-９,，.．%％~〜+±▲△一-九十百千万億兆年月日円人件台時分秒回歳割約-]+$
    • 年号: ^(令和|平成|昭和)[0-9０-９元一-九十]*(年度?|年)?$
    • 西暦四半期: ^[0-9０-９]{4}年?[0-9０-９]*月?(期|度)?(第[0-9０-９一-四]四半期)?$
  2. ② merge_duplicate_id_nodes — 型分裂マージ
    同一 id が複数ラベルに分裂（LLMGraphTransformerが (id,type) 単位でノード生成するため）。次数最大のノードをkeeperに、他を _move_rels で寄せる。extraction_countを合算、source_chunksを和集合、ラベルを追加。実測2,600 id / 5,557ノード。
  3. ③ merge_kana_variant_nodes — かな揺れマージ
    送り仮名・助詞・長音のみ違うノードを統合（ガス軸受/ガス軸受け、サーバ/サーバー）。kana_variant_key 骨格キーでグループ化 → union-findで連結成分 → keeperに寄せ、dup idを aliases に追加。詳細は下表。
  4. ④ normalize_relations — 関係正規化（27ルール）
    逆関係・同義・typoを正規タイプへ統合。検索Cypherは無向マッチのため逆関係は情報量ゼロで重複とextraction_count分散だけ生む。batch_size=2000。詳細は下表。
  5. ⑤ flag_anaphor_nodes / resolve_anaphora_nodes — 照応処理
    「本製品」「当社」等の照応ノードを解決 or is_anaphor=true フラグ（ハード除外: 起点・中継・終端すべて禁止）。参照グラフ由来の alias_maps を使う（§5と連携）。

かな揺れ判定 _is_kana_variant_pair(a,b,max_diff=3)

  • どちらか空・等しい・先頭文字が違う → False（この文書/文書 を拒否）
  • 末尾 ー を除去して等しければ True（サーバ/サーバー）
  • difflib.SequenceMatcher で差分文字を収集
  • 差分が 3 文字超 → False
  • 差分が全てひらがなのときのみ True（漢字/数字/英字の差は却下 → U7314/U7414、第3/第4四半期、取締役/監査役を保護）

骨格キー kana_variant_key japanese.py:36-55

  • 空白・・ を除去、NFKC正規化
  • 末尾長音 ー を除去（語中は音価があるため保持）
  • ひらがなを全除去して骨格化
  • 漢字/カタカナが無ければ ""（型番 U7314 を保護）
  • len(skel)<3 または len(skel)*2 ≤ len(t) なら ""（わかりやすい情報→情報 のような語の消失を拒否）

④ 関係正規化 27ルール RELATION_NORMALIZE_MAP consolidate.py:305-336

マッピングは {旧: (新, 方向反転)}。負関係（NOT_*/EXCLUDES/DOES_NOT_HAVE）は意図的に除外。

POST6. 条件付き事実（reify）graph/conditions.py

既定OFF 規程・基準系限定。「ある値がある条件下でのみ成立する」事実を :CondFact -[:WHEN]-> :Cond のreify構造で格納。数値は決してノード化せず value_num プロパティに。

データモデル（reification）

:CondFactlabel / predicate(UPPER_SNAKE 例:WAGE_PREMIUM_RATE) / value_num(例:35) / unit(%/日/円) / source / pg_collection / source_chunks / extraction_count
▼ [:WHEN]（複合ANDは複数エッジ）
:Condaxis(労働区分/時間帯/閾値…) / operator(eq/gte/lte/gt/lt/between/in) / value(逐語) / norm_value / pg_collection

抽出フロー

  1. プレフィルタ: len(text)≥120 かつ _COND_KEYWORDS（場合/とき/以上/割増/手当/勤続…）に一致
  2. structured_output(llm, Extraction) で JSON 抽出（ThreadPoolExecutor(max_workers=4)）
  3. 条件のない事実は破棄（無条件の単純事実は対象外）

POST8. enrichment（プロパティ付与）graph/enrichment.py

enrich_post_build(graph) が3つのパスを実行。検索の順位付けと照合キーの基盤を作る、最重要の後処理。

A. compute_mention_count

t.mention_count = COUNT(DISTINCT d)（(t)<-[:MENTIONS]-(d:Document)）。何チャンクから言及されたか。

B. compute_pagerank alpha=0.85, max_iter=100

APOC不要、NetworkX使用。MENTIONS除外・両端がentity述語を満たすエッジで nx.DiGraph を構築 → nx.pagerank → t.pagerank をbatch 500で書き戻し。起点ノード選択とパススコアの主信号。

C. compute_search_keys retrieval照合の要

辞書適用の後に走る。各エンティティに norm_id と search_keys 配列を書く（batch 2000）:

  1. normalize_entity_text(id)（NFKC+小文字化）
  2. 各 aliases の正規化
  3. canonical_form の正規化
  4. 各キーの kana_variant_key（骨格キー）
順序保持dedup。norm_id=keys[0]。検索側は search_keys に対し CONTAINS 照合し、全角半角・大小文字・送り仮名・助詞・長音の揺れを吸収。

D. attach_source_chunks（チャンク処理中にインライン実行）

各トリプルの抽出元チャンクIDを edge.source_chunks に冪等追加、extraction_count をインクリメント。MERGE でなく MATCH を使う（同名別ラベルノード間の誤エッジを避ける）。これが検索側「① source_chunks 直引き」を可能にする。

正規化エンティティを起点に、Neo4jでN-hopパスを「スコア順」でそのまま取得する。ホップ数ごとに取得数・リランク後件数を変える。

ホップ別パラメータ

ホップ数Cypher LIMIT（limit_map）リランク後 top_k（top_k_map）

  1. 起点ノード選択 _start_select
    search_keys（未バックフィル時は toLower(n.id)）に対し CONTAINS 照合。完全一致を最優先し、次に pagerank 降順で上位[0..20]に絞る。旧実装は無作為collectで汎用語が起点を占有していた。
  2. パス取得（ノイズ除外つき）
    1-hop は (start)-[r]-(end)、N-hop は (start)-[*1..hop]-(end)。除外条件:
    • ハッシュIDノード（チャンク）を除外
    • MENTIONS エッジを除外
    • is_anaphor ノードは起点・中継・終端すべて禁止
    • is_value ノードは中継禁止・終端のみ許可（日付ハブ経由の逸脱を防ぎつつ、数値の出典チャンク取得は残す）
  3. スコアリング（無作為truncation排除）
    1-hop: extraction_count DESC, pagerank和 DESC。
    N-hop: パス長で正規化した平均 rel_sum/plen DESC, pr_sum/(plen+1) DESC（長さ自体には加点しない）。
  4. parse_neo4j_paths
    (ノード列, 関係タイプ列) で重複排除。Cypher側スコア順を保持（パス長降順ソートは廃止 = 1-hop事実エッジが落ちる逆効果のため）。max_candidates=30。

    len(paths) > top_k のときのみ rank_paths_by_relevance。以下ならスキップ。§14

RETRIEVAL14. リランキング（cross-encoder優先）retrieval/reranker.py

全レイヤ（文書・パス・トリプル・KGチャンク）で共通の2段構成: 広く取得 → cross-encoderで意味的に絞る。cross-encoder失敗時はLLMリランクにフォールバック。

score_candidates(query, candidates) vLLM /v1/score（bge-reranker-v2-m3互換）。payload {model, text_1:query, text_2:[候補]}。失敗・長さ不一致 → None（=フォールバック信号）。
rerank_by_score(query, items, text_fn, top_k) 汎用ヘルパ。text_fnで各itemからテキスト抽出 → スコア降順 → top_k。None時は items[:top_k]（原順）。

パス/トリプルの「自然文化」

cross-encoderは自然文ペアで学習されているため、記号表記 A -[IS_A]-> B はスコアが不安定。_REL_TEMPLATES（39関係）で自然文に変換してからスコアリングする（表示用 path_text は記号表記のまま）。

ガス軸受 -[IS_A]-> 軸受 → triple_to_natural_text → 「ガス軸受は軸受の一種である」
未知関係は 「{s}と{o}は{rel}の関係にある」。doc_contextはcross-encoderには連結しない（信号が薄まるため）が、LLMフォールバック側では使う。

文書リランク rerank_with_llm

  • cross-encoder優先: text_fn=d.text[:1000]（500字だと後半の根拠が隠れるため1000字）
  • LLMフォールバック: text[:500] + RERANK_PROMPT → 「3,1,5,2,4」形式のID列パース → 欠落IDは末尾追加

有効化ゲート

is_reranker_enabled() = RERANKER_ENABLED=true かつ endpoint設定あり。
速度: cross-encoderはLLMリランクの10〜100倍高速。品質寄与 +11.8pt（最強レバー）のため既定ON。

GENERATION16. 回答生成 ui/system.py :: generate_with_sources

統合コンテキストを QA_PROMPT に流し、temperature=0 のLLMで回答生成。LangChain LCELチェーンで構成。

question→ retriever_and_merge→ {context, question}→ QA_PROMPT→ LLM(temp=0)→ StrOutputParser→ answer + sources

QA_PROMPT の核心ルール

Grounding（根拠限定）
  • 参考情報の範囲「だけ」で回答。外部知識・推測で補完しない
  • 断定文には必ず根拠（文書 or トリプル）を添える
  • 規格名・JIS番号・条文番号は一字一句転記（反捏造）
  • 矛盾する情報は両論併記し、どちらとも断定しない
コンテキストの読み方
  • <GRAPH_CONTEXT>: 推論パス（マルチホップ）と単体トリプル。番号は関連度順
  • <DOCUMENT_CONTEXT>: 原文チャンク。事実・数値・条件の根拠は基本こちら
  • <CONDITION_FACTS>: 条件→値の機械的列挙（あれば最優先で先頭）
出典書式（inline citation）: 文書は [出典: ファイル名]、グラフは [出典: KG トリプルN] / [出典: KG 推論パスN] / [出典: KG PN-Ej] をラベル通りに。規程・条文系では「何に準拠/準用するか」を具体値より先に明示。
